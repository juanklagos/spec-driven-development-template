**SDD Desk** is the visual builder in its own window. It carries its own Node runtime, so
nothing needs to be installed first, and it hosts the SDD server itself — including the MCP
endpoint your agents connect to.

<Aside type="caution" title="Your system will warn you the first time">

[what that actually means](#why-that-warning-appears) and
[how to open it](#opening-it-the-first-time) on each system.
</Aside>

## Why that warning appears

Signing a program and publishing it under an open licence are two different things, and they
are easy to confuse.

**The licence says what you may do with the code.** SDD Desk is MIT: use it, read it, modify
it, redistribute it, sell it. All of the code is in the repository, including the code that
builds these downloads.

**The signature says who assembled the file you downloaded.** macOS and Windows do not read
licences: they check whether a paid certificate vouches for the identity of whoever signed the
binary. An unsigned MIT `.dmg` and an unsigned proprietary one get exactly the same warning.

So the warning is **not** telling you the app is broken, or dangerous, or that your system
detected something. It says, literally, that the system cannot verify who published it. Whether
it is reasonable to go ahead depends on whether you trust the source — which is what having the
code in the open is for.

Why it is not signed yet: Apple charges USD 99 a year with no exception for open source.
Windows does have a free route for OSI-approved licences, which MIT satisfies. Neither changes
what the app does or who may use it — only whether the dialog appears. Linux asks for no
signature at all.

## Opening it the first time

<Tabs syncKey="os">
  <TabItem label="macOS" icon="apple">
    Open the `.dmg` and drag **SDD Desk** to Applications. Then, in the Applications folder,
    **right-click the app and choose Open** — that dialog has an Open button, which the one from
    a double click does not. Only needed the first time.

    <Aside type="danger" title="If it says “is damaged and can’t be opened”">
    That is **not** the normal warning, and it does not mean the download arrived corrupted. It
    was a packaging bug in 0.1.1: the bundle shipped without a valid code signature seal, and
    Apple Silicon rejects that outright instead of asking for confirmation. Fixed in 0.1.2 —
    download again and it will not happen. To rescue the copy you already have:

    ```bash
    xattr -dr com.apple.quarantine "/Applications/SDD Desk.app"
    codesign --force --deep --sign - "/Applications/SDD Desk.app"
    ```
    </Aside>

    If instead you see that the developer "cannot be verified", that one is the expected warning
    and right-click → Open resolves it. You can also clear the quarantine flag by hand, only on
    a download you trust:

    ```bash
    xattr -dr com.apple.quarantine "/Applications/SDD Desk.app"
    ```
  </TabItem>
  <TabItem label="Windows" icon="seti:windows">
    SmartScreen will show "Windows protected your PC". Click **More info**, then
    **Run anyway**. The warning appears because the installer is unsigned, and it will keep
    appearing until the project has a certificate.
  </TabItem>
  <TabItem label="Linux" icon="linux">
    Make the AppImage executable and run it:

    ```bash
    chmod +x SDD-Desk-linux.AppImage
    ./SDD-Desk-linux.AppImage
    ```

    A `.deb` is published alongside it for Debian and Ubuntu. There is no Flatpak: Flathub has
    banned AI-assisted code since 2026-05-29.
  </TabItem>
</Tabs>

## Two routes with no warning at all

The app is MIT, like the rest of the project. If you would rather not deal with Gatekeeper or
SmartScreen, there are two ways to use exactly the same thing without any system warning you
about anything.

**The builder in your browser.** One command, nothing to download:

```bash
npx @juanklagos/sdd-mcp@latest --http
```

Same canvas, same server, same MCP endpoint. The only things it does not give you are the
window of its own and the Dock icon — and it needs Node installed, which is precisely what the
desktop app exists to avoid.

**Build the app yourself.** A binary you compiled on your own machine did not come from the
internet, so nothing quarantines it:

```bash
git clone https://github.com/juanklagos/spec-driven-development-template.git
cd spec-driven-development-template && npm ci && npm run build
cd builder && npm ci && npm run build && cd ../desk
npm install && npm run dist
```

The result lands in `desk/release/`. Node is needed only to build it; the resulting app is not.

## What happens when you open it

1. It asks for your project folder — one that has a `specs/` directory or a `spec/` sidecar.
   Don't have one? Create it with `npx @juanklagos/create-sdd-project@latest .`
2. The board opens on that project. Your files on disk stay the source of truth; the app never
   copies them anywhere.
3. It remembers the folder. **File → Open project…** switches to another one without a restart.

## Connecting an agent

The app *is* your project's MCP server while it is open. **File → Connect an agent…** shows the
URL and a config snippet you can copy:

```json
{ "mcpServers": { "sdd": { "url": "http://127.0.0.1:3334/mcp" } } }
```

The port is whatever the app actually bound — it moves to the next free one if 3334 is taken,
which is why the dialog shows you the real number instead of you assuming it.

Anything your agent writes through that endpoint appears in the window immediately, without a
reload.

## What it does not do

It runs entirely on your machine. There is no account, no sync and no server of ours involved —
the app opens a port on `127.0.0.1` that nothing outside your computer can reach.

It is also not a replacement for the terminal route. `npx @juanklagos/sdd-mcp@latest --http`
still opens the same builder in your browser, and still works exactly as before.

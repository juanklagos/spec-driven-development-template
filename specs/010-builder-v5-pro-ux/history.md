## 2026-07-21 (seguimiento: el dashboard y la regla de estado)

- El autor señaló que `/dashboard` había quedado "muy básico" frente al builder ya migrado a shadcn. Rediseñado con un panel de jueces de diseño (3 propuestas independientes → evaluación → síntesis): capa ejecutiva, banda de compuerta, KPIs, lista de specs con progreso, avisos de dependencias, tokens exactos del builder, dark/light, y un solo idioma (`?lang` → `Accept-Language` → es) con toggle ES|EN.
- Hallazgo durante la verificación: la regla de estado de una spec estaba duplicada en cuatro sitios y **divergía**. `SpecNode` usaba `/aprobado|approved/i` (no reconocía "Aprobada"), el kanban `/aprobad[oa]|approved/i`, el drawer una tercera copia para habilitar "Implementar con agente", y el dashboard exigía aprobación para "hecha". Consecuencia real: una spec "Aprobada" salía pendiente en el lienzo pero aprobada en el kanban; y una spec con todas las tareas marcadas sin aprobar salía "Hecha" en el lienzo y "Pendiente" en el dashboard.
- Corregido en la fuente única: `specTone` en `sdd-core` (aprobación primero — sin aprobación no hay "hecha", que es justo el anti-patrón que el template existe para exponer), expuesto como `BoardSpecCard.tone` por la API REST y las tools MCP; lienzo, kanban, drawer y dashboard solo lo pintan. Único espejo documentado: `sections.ts` para la pestaña que lee el markdown crudo.
- Verificado con los dos casos borde en un workspace real: API, dashboard, tarjeta del lienzo y columna del kanban coinciden (capturas). Asserts de regresión añadidos a `mcp:test`; el esquema de salida de las tools MCP incluye `tone`.
